# v14.10.0 — Power Damage, Self-Damage & Test Suite Recovery

**Release Date**: July 2026

## Major Features

### Power Damage

Action cards can now deal **Power damage** alongside regular (Resolve) damage in a single move. Power damage targets the actor's **Power** resource instead of Resolve, enabling characters that drain or restore power as part of their attacks.

**What's New**

- **Power Damage Fields** — Both Attack Chain and Saved Damage modes now have a Power Damage configuration block (formula, type, and — for Attack Chain — its own independent condition and threshold). The power damage formula defaults to `0`, meaning no power damage is applied unless configured.
- **Independent Conditions** — In Attack Chain mode, power damage has its **own** condition (defaulting to *One Success*), separate from the resolve damage condition. This means power damage can fire even when the resolve damage condition is not met (for example: resolve damage on two successes, power damage on one).
- **Smart Card Routing** — When both resolve and power damage apply to a target, they render in a **single combined stacked chat card** with two roll sections, each with its own attached roll and section label ("Resolve Damage" / "Power Damage"). When only one damage type applies, the **classic single tall card** is used instead — no layout change from previous behavior.
- **Purple Styling** — Power damage and power heal are visually distinguished with a purple theme (headers, value text, and section labels). Numbers on stacked cards are 10% smaller to fit both sections comfortably.
- **Power Heal & Overhealing** — Power damage supports the `Heal` type (restores power instead of draining it). When a power heal exceeds the target's maximum power, an overcharge indicator is shown, mirroring the existing resolve overhealing display.
- **Power Damage Per Success Toggle** — A new config toggle controls whether power damage applies on every repetition or just the first, parallel to the resolve damage toggle.

### Renamed: "Resolve Damage Per Success"

The existing **Damage Per Success** toggle has been renamed to **Resolve Damage Per Success** to distinguish it from the new **Power Damage Per Success** toggle. The two toggles sit side by side in the action card config tab.

**What Changed**

- `DamageApplication` label changed from "Damage Per Success" to "Resolve Damage Per Success"
- New `powerDamageApplication` toggle (defaults to enabled, matching resolve damage)

### Repetition Awareness

Power damage is treated as a first-class damage type in repetition logic:

- The **Fail on First Miss** check now considers power damage — if the first repetition's power damage condition is met, the chain is considered successful and continues, even if resolve damage did not trigger.
- Each damage type's per-repetition toggle (`Resolve Damage Per Success` / `Power Damage Per Success`) is evaluated independently.

### Self-Damage / Self-Heal

Action cards can now apply **damage or healing to their own owner** (the card's actor), in addition to the existing self-status-effects system. This supports mechanics like recoil, self-sacrifice, or self-healing on a hit.

**What's New**

- **Self-Damage Configuration** — A new config block on action cards (attack-chain mode) with an independent condition (defaulting to *Never*), threshold, and separate Resolve and Power formula/type pairs.
- **Resolve & Power** — Self-damage can affect the owner's Resolve, Power, or both simultaneously. Each supports the `Damage` (reduce) and `Heal` (restore) types.
- **Condition-Based** — Mirrors the self-effects pattern: the condition is evaluated against the aggregated target hit results, so self-damage can trigger on *One Success*, *Always*, *Zero Successes*, or any other condition.
- **Smart Card Routing** — Self-damage uses the same card selector as target damage: single tall card for one type, stacked card for both.
- **Defaults to Never** — No behavior change for existing cards until the condition is changed from *Never*.

### Test Suite Recovery

The entire test suite has been recovered from **197 failing tests to 0**. All 3385 tests now pass.

**What Was Fixed**

- **Roll mock** (`tests/setup.mjs`) — The mock now evaluates formulas with `@data` references, math functions (`max`, `floor`, etc.), dice terms, and a working `evaluateSync()` method. Previously it returned a hardcoded `total: 10` with no `evaluateSync`.
- **Schema defaults** — Fixed the `TypeDataModel` mock's nested schema initialization, which was flattening nested defaults (e.g., `abilities.acro.multiplyNeutral`) to the top level instead of nesting them properly. This was the root cause of 33+ `base-actor` test failures.
- **`foundry.utils`** — Added working implementations for `getProperty`, `setProperty`, `mergeObject`, `randomID`, `deepClone`, `expandObject`, `flattenObject`, `Collection`, and more. The `@rayners/foundry-test-utils` library provided empty `vi.fn()` stubs that returned `undefined`.
- **`CONST`** — Added `ACTIVE_EFFECT_SHOW_ICON` (`NEVER`/`CONDITIONAL`/`ALWAYS`) and `TOKEN_DISPLAY_MODES` mocks.
- **`CONFIG`** — Added baseline `EVENTIDE_RP_SYSTEM.abilities` and `hiddenAbilities` configuration.
- **20+ test files updated** for v14 API changes: ActiveEffect mode strings, context-menu property renames (`name`→`label`, `callback`→`onClick`), `rollMode`→`messageMode`, migration method renames, stale count assertions, and more.

### CI/CD Pipeline

Tests and coverage reporting are now **re-enabled** in the GitHub Actions workflow. All pull requests to `main` and `develop` will automatically run linting, the full test suite, and upload coverage to Codecov.

## Migration Notes

No migration required. All new schema fields use `initial` values that auto-populate on existing action cards:

- Power damage formulas default to `0` (no power damage until configured)
- Self-damage condition defaults to `Never` (no self-damage until configured)
- `powerDamageApplication` defaults to `true` (matching resolve damage)
- The `DamageApplication` rename is display-only — the underlying field key (`damageApplication`) is unchanged
- The test suite recovery is infrastructure-only — no production behavior changes
