# v14.5.0 - Configurable Restore Targets & Per-Target Self-Effect Scaling

**Release Date**: May 2026

## Major Features

### Per-Target Self-Effect Scaling for Action Cards

Action cards now support a **Scale Self-Effects Per Target** toggle that enables self-effects to be applied once for each target that meets the success condition, rather than once total. This supports use cases like a vampire's mass drain attack that should grant 1 fortitude per successful target hit.

**What It Does**

A new toggle — **Scale Self-Effects Per Target** — has been added to the action card Config tab under the Behavior section. When enabled (default: off), the self-effects condition is evaluated independently for each target, and self-effects are applied to the card owner once per qualifying target. The existing **Self-Effects Application Limit** still caps the total number of applications across all targets.

**How It Works**

1. On an action card's Config tab, enable **Scale Self-Effects Per Target**
2. Set the **Self-Effects Application Limit** to `0` (unlimited) to allow one application per qualifying target, or set a specific cap
3. Configure your self-effect condition (e.g., "On One Success") as usual
4. When the action card hits multiple targets, each target that meets the condition triggers a separate self-effect application on the caster

**Use Cases**

- **Mass drain** — A vampire targets 5 enemies with a drain attack. Each enemy hit grants the vampire +1 fortitude, for a total of 5 applications
- **Capped scaling** — Same scenario, but with application limit set to 3. Even if 5 enemies are hit, only 3 fortitude applications occur
- **Default behavior preserved** — With the toggle off, self-effects apply once total regardless of how many targets are hit (existing behavior unchanged)

**Behavior Details**

| Scenario | Scale Per Target | Application Limit | Targets Hit | Self-Effect Applications |
|----------|-----------------|-------------------|-------------|--------------------------|
| Default (existing) | Off | 1 | 3 | 1 |
| Per-target scaling | On | 0 (unlimited) | 3 | 3 |
| Per-target scaling with cap | On | 2 | 5 | 2 |
| No targets hit | On | 0 | 0 | 0 |

**Important Notes**

- The **Self-Effects Application Limit** caps the total number of applications regardless of how many targets trigger them
- When the toggle is off, the condition is evaluated once against aggregated results (at least one target hit = `oneSuccess`, all targets both-hit = `twoSuccesses`)
- When the toggle is on, the condition is evaluated per-target (each target's own hit results determine if the condition is met)
- No migration is required — the toggle defaults to `false`, preserving existing behavior

**Schema Changes**

The action card data model now includes one new field:

| Field | Default | Description |
|-------|---------|-------------|
| `system.selfEffectsScalePerTarget` | `false` | When true, self-effects scale per qualifying target instead of once total |

### Power and Resolve Restore Targets

Actors now support configurable **restore target** values for power and resolve, allowing the Restore Target macro to restore resources to a custom value instead of defaulting to max. This enables character concepts where "restored" means something other than full — for example, a character that starts at 0 power and gains power as they use abilities.

**What It Does**

Two new override abilities — **Power Restore Target** and **Resolve Restore Target** — can be configured as Settings Effects on items. When set on a character, the Restore Target macro will restore the resource to that specific value instead of its maximum. If no restore target is set (the default), the macro behaves exactly as before, restoring to max.

**How It Works**

1. On any item's Character Effects tab, open the **Settings Effects** section
2. Add a **Power Restore Target** or **Resolve Restore Target** effect
3. Set the value to the desired restore target (e.g., `0` for a character that starts at 0 power)
4. When the GM runs the **Restore Target** macro on that character, the resource will be restored to the configured target value instead of max
5. The restore target value is clamped between `0` and the character's max, so it cannot exceed normal bounds

**Use Cases**

- **Power-from-zero characters** — Set Power Restore Target to `0` so the Restore macro resets power to 0, matching the character's baseline state
- **Partial recovery** — Set a restore target below max for characters who should only partially recover (e.g., a character with max 20 power but a restore target of 10)
- **Full override of max** — Set a restore target equal to the override max for characters whose max has been modified
- **Default behavior preserved** — Characters with no restore target set (null, the default) restore to max as always

**Behavior Details**

| Scenario | Restore Target | Behavior |
|----------|---------------|----------|
| Default (most characters) | `null` | Restore to max (unchanged behavior) |
| Power-from-zero character | `0` | Restore sets power to 0 (clamped to `[0, max]`) |
| Partial recovery | Any number | Restore sets resource to that value (clamped to `[0, max]`) |
| Target above max | Any number | Restore sets resource to max (clamped by max) |

**Important Notes**

- Restore targets are **set via Settings Effects** on items, the same way Power Override and Resolve Override are configured
- The restore target only affects the **Restore Target macro** — healing rolls and other resource changes are unaffected
- Restore targets use the same nullable integer schema pattern as overrides, defaulting to `null`
- When a restore target is set, the value is **set directly** (not added), unlike the default behavior which adds max to the current value

**Schema Changes**

The actor data model now includes two new nullable fields:

| Field | Default | Description |
| ----- | ------- | ----------- |
| `system.power.restoreTarget` | `null` | When set, the Restore macro sets power to this value instead of max |
| `system.resolve.restoreTarget` | `null` | When set, the Restore macro sets resolve to this value instead of max |

## Bug Fixes

### Action Card Config Tab Scroll Jump Fix

Toggling checkboxes in the action card Config tab while scrolled down caused the scrollable container to shrink and the top of the sheet to be cut off. The custom `ScrollPreservationMixin` used a fragile approach of scanning all DOM descendants for `scrollTop > 0` to find the scrollable element, then reconstructing a CSS selector from its class names to relocate it after re-render. This could match the wrong element or fail entirely if classes changed between renders. Additionally, scroll restoration was applied synchronously before `EditModeCheckMixin`'s delayed `setTimeout(10ms)` DOM modifications (disabling/enabling inputs) had settled, causing layout shifts after restoration.

**What Was Fixed**

- **Removed custom mixin entirely**: Replaced `ScrollPreservationMixin` with Foundry V2's built-in `HandlebarsApplicationMixin` scroll preservation via `scrollable: [""]` on all tab-content PARTS in both `EventideRpSystemItemSheet` and `EmbeddedItemSheet`
- **HTML nesting fix**: Removed an extra `</div>` in `action-card.hbs` that prematurely closed the `eventide-sheet__scrollable-container`, breaking scroll containment
- **CSS stabilization**: Added explicit `display: block; width: 100%` on `.eventide-sheet__scrollable-container` to prevent Foundry's flex layout from causing content shifts during re-renders
- **ESLint cleanup**: Removed `requestAnimationFrame` and `cancelAnimationFrame` from ESLint globals (no longer needed)

### Known Issues

- **Minor scroll shift on Config tab toggle**: Toggling the "Scale Self-Effects Per Target" checkbox while scrolled down on the action card Config tab may cause a ~20px upward content shift. Switching tabs resets the layout. This is a Foundry V2 framework-level timing issue in the re-render cycle and does not affect data or functionality.

### Advantage/Disadvantage Had No Effect on Ability Rolls

Character effects granting advantage or disadvantage on ability rolls were silently ignored. The `buildDiceFormula()` utility requires an `absTotal` property on the dice adjustments object to determine how many dice to roll, but the actor and item data model schemas only defined `advantage`, `disadvantage`, `total`, and `mode` — no `absTotal`. Since `undefined > 0` evaluates to `false`, the function always fell through to rolling a single die regardless of advantage or disadvantage values.

**What Was Broken**

- A character with advantage 1 on an ability would still roll `1d20+X` instead of `2d20kh1+X`
- A character with disadvantage would also roll `1d20+X` instead of `2d20kl1+X`
- Item rolls were unaffected because they pass through `combineDiceAdjustments()` which correctly computes `absTotal`

**What Was Fixed**

Added the missing `absTotal` field to the `diceAdjustments` schema and computed it as `Math.abs(total)` in `prepareDerivedData()` across all four data models that define dice adjustments:

| Data Model | Change |
|------------|--------|
| `base-actor.mjs` | Added `absTotal` field to ability dice adjustments schema + computation |
| `item-combat-power.mjs` | Added `absTotal` field + computation |
| `item-feature.mjs` | Added `absTotal` field + computation |
| `item-gear.mjs` | Added `absTotal` field + computation |

## Migration Notes

No migration is required for this release. The `selfEffectsScalePerTarget` field defaults to `false`, preserving existing self-effect behavior. The `restoreTarget` fields default to `null` for all existing actors, preserving the existing restore behavior (restoring to max). Existing saves and data are fully compatible.
